# User Story 整理技能

## 技能定位

将**零散的需求诉求**转化为**结构化的用户故事列表**，供团队讨论和任务分配。

**不做的事**：
- 不落盘，不生成文件
- 不写验收标准（AC）
- 不做架构方案设计
- 不拆 Issue / 不生成 Plan

---

## 输入规则

`$ARGUMENTS` 可以是以下两种之一，也可以两者结合：

1. **口述/粘贴的需求要点**：用户直接描述若干条诉求，格式不限（草稿、流水账、会议纪要均可）
2. **草稿文件路径**：指向仓库内任意可读文件，技能主动读取

未提供任何输入时：停止推进，询问用户提供需求要点或文件路径。

---

## 执行阶段

### 阶段 A：输入解析

1. 若有文件路径，先读取文件内容
2. 将输入内容解析为若干需求诉求条目
3. 识别条目之间的关联关系（共享数据流、互相依赖、同一交付面等），记录备用
4. **合并判断**（关键步骤）：对解析出的条目做一轮合并检查，满足以下任一条件时应合并为一条故事：
   - 落在**同一功能页面/同一交互区域**（如同一 Dashboard 的不同图表、同一表单的不同字段）
   - 共享**同一数据源或计算逻辑**（如概览卡 + 折线图都消费同一组聚合指标）
   - 属于**同一用户角色的连续操作流**（如配置 + 查看 + 接收通知）
   
   合并后的故事在「期望能力」中拆点列出各子能力，保持用户视角的完整性。**宁可 3 条丰满的故事，不要 8 条单薄的故事。**

### 阶段 B：代码背景调研（主动执行）

输出用户故事前，先调研项目现有代码，目的是：
- 准确描述**当前痛点**（现在怎么做、现在缺什么）
- 避免将已有能力误写为待新增
- 让「期望能力」与系统现状对齐

**调研策略**：
1. 从需求诉求中提取关键词（模块名、功能名、字段名、流程名）
2. 用 Grep / Glob 在仓库中定位相关文件（模型、服务、接口、配置等）
3. 重点读取与本次需求直接相关的文件，不做全量扫描
4. 调研深度以「能写出准确的当前痛点」为止，不需要深入到架构设计层

调研结果**不向用户汇报**，直接用于撰写用户故事。调研过程中的技术发现（文件路径、函数实现、字段定义等）**只作为内部判断依据，严禁直接出现在用户故事输出中**。

若调研后发现某条诉求对应的能力**已存在**，在「当前痛点」中以用户视角描述当前体验（如「目前无法在页面上完成…」），「期望能力」只描述差距部分。

### 阶段 C：输出用户故事

按合并后的故事条目输出，使用统一格式。全部故事输出后，附上「故事间关系」节。

---

## 输出格式

### 单条用户故事

```
### US-N：<故事标题>　【优先级：P0 / P1 / P2】

**用户故事**
作为一名 **<目标用户角色>**，我希望 <能做什么>，以便 <达成什么目标>。

**当前痛点**
<基于代码调研，具体描述系统现状、缺失点或操作成本。>

**期望能力**
- <具体能力点 1>
- <具体能力点 2>
- <...>
```

> **合并故事写法**：当一条故事聚合了多个交付面（如概览卡 + 趋势图），标题用「数据展示」「管理能力」等概括性表述，「期望能力」拆点列举各子能力（如「概览区展示 ××」「趋势图展示 ××」）。用户故事正文保持一句话概括整体目标。

### 优先级判断标准

| 优先级 | 含义 | 判断标准 |
|--------|------|----------|
| P0 | 核心阻塞 | 没有这个能力，某类场景完全无法跑通 |
| P1 | 重要增强 | 有替代方案但效率低，或影响结果准确性 |
| P2 | 体验优化 | 不影响核心流程，锦上添花 |

优先级由技能根据需求描述和代码现状自行判断，用户可在输出后调整。

### 故事间关系（全部故事之后）

```
## 故事间关系

**数据/逻辑联动**：
<描述多条故事之间的输入输出关系或共享机制>

**依赖关系**：
- US-X 需先于 US-Y 实现（X 的能力是 Y 的前提）
- US-A 与 US-B 可并行开发

**可合并建议**（如有）：
- US-C 与 US-D 范围高度重叠，可讨论合并
```

---

## 输出原则

1. **准确优先**：当前痛点必须来自代码调研，不凭空描述
2. **聚焦差距**：期望能力只写「缺什么」，不重复描述已有能力
3. **优先合并，颗粒饱满**：同一交付面（页面/数据源/操作流）的能力尽量合并为一条故事，避免按"指标卡 ×1、图表 ×1、告警 ×1"机械拆分。目标产出 2~4 条故事，而非 6~8 条
4. **不预设方案**：故事描述「要什么」，不描述「怎么实现」
5. **角色真实**：根据实际使用场景选择用户角色，不套用固定模板
6. **严禁代码细节渗入输出**：用户故事面向真实用户（管理员、运营、最终使用者等），禁止出现以下内容：
   - 文件名、类名、函数名、路由路径（如「在 xxx.py 中」「调用 xxx 接口」）
   - 数据库字段名、ORM 字段、表名（如「user 表的 status 字段」）
   - 实现步骤、代码改动描述（如「需要在 service 层新增」「前端要修改 xxx.vue」）
   - 技术选型或框架细节（如「用 Redis 缓存」「通过 WebSocket 推送」）
   
   **代码调研是内部手段，用于理解现状；输出只表达用户视角的感受与诉求。**

---

## 多次调用规则

- **首次调用**：完整执行 A → B → C
- **「调整第 N 条」**：仅修改对应故事，重新输出完整列表
- **「加一条」**：追加新故事，更新「故事间关系」
- **「改优先级 / 换角色」**：直接修改对应字段，重新输出
