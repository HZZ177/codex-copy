# Requirement Clarify（需求澄清）

## 目标

通过 `/req` 或 `/req --simple` 命令，将用户提供的草稿目录或文字描述整理成规范化的需求文档（REQ），保存到 `.ktaicoding/req/` 目录。文档可作为后续 `/design` 流程的输入。

## 交互节奏（强制）

- **一轮一步**：每一轮对话只推进下面「第二阶段」中的**一个** Step（`--simple` 可合并步骤，但仍须遵守「一轮一步」，不得在同一轮里跨 Step 追问）。
- **轮次**：标准版按下方 Step 表推进，强引导不少于 **7 轮**（R1–R5 与 R8 为必做；R6/R7 可选各占一轮或跳过）；**不**为省轮次合并多个强引导主题。
- **本轮产出边界**：本轮只输出当前 Step 的说明、摘录或追问；**禁止**在同一轮附带其他 Step 的完整问卷或整份 REQ 结构。
- **结束语**：每轮末尾用固定语义停顿——信息不足则 **≤5 条** 追问；已收齐则写「本步已完成。回复 **继续** 进入下一步；或 **修订：…** 先改本步。」
- **落盘方式（用户选择）**：全部必做 Step 完成后，**不得默认写文件**。先询问用户：**「直接落盘」**（按模板写入 `.ktaicoding/req/`）或 **「先生成预览」**（仅输出完整或摘要级草案，用户确认后再落盘）。用户明确说「直接落盘 / 写入 / 确认生成文件」之一后，再执行写入。

## 触发方式

- **标准版**: `/req` — 生成完整的需求文档
- **简化版**: `/req --simple` — 生成简化版需求文档，适用于小需求

## 输入要求（强制）

**禁止自动扫描仓库**。用户必须显式提供以下之一：

1. **需求草稿目录**：用户指定的目录路径，技能仅读取该目录下用户指明的文件
2. **需求文字描述**：直接在对话中给出的 PRD/要点/口述内容

**未提供输入时**：提示用户补充「草稿目录」或「文字描述」，不得猜测或扫描仓库。

## 文档命名规则

### 格式
```
REQ-{YYYYMMDD}-{序号}-{中文名称}.md
```

- `{YYYYMMDD}`：8 位日期（公历）
- `{序号}`：当日流水序号，用户显式指定；未指定时默认 `001`，若已存在则提示用户指定新序号
- `{中文名称}`：需求中文简称，做安全文件名处理（小写/连字符 slug、去非法字符），限制在10字以内

### 输出文件
- **标准版**: `.ktaicoding/req/REQ-{YYYYMMDD}-{序号}-{中文名称}.md`
- **简化版**: `.ktaicoding/req/REQ-{YYYYMMDD}-{序号}-{中文名称}-简化版.md`

### 文件名冲突
若已存在相同文件名，提示用户调整序号或名称，禁止静默覆盖。

## 工作流程

### 第一阶段：获取输入

1. 确认用户是否提供了草稿目录或文字描述
2. 若未提供，提示用户补充
3. 解析已有内容，映射到模板章节
4. 标记空缺项

### 第二阶段：引导完善（强制 · 单轮单步）

对空缺、模糊、占位内容进行分轮交互引导。**每个 Step 独占一轮对话**；话术示例见本文「引导话术示例」，**禁止**把多个示例块在同一轮全部贴给用户。

| Step | 本轮主题 | 完成标记 |
|------|----------|----------|
| R1 | 用户故事 | 用户确认或修订 |
| R2 | 目标用户 | 用户确认或修订 |
| R3 | 需求背景（业务背景、问题、预期收益） | 用户确认或修订 |
| R4 | 功能与流程（主路径 + 关键异常分支） | 用户确认或修订 |
| R5 | **本期边界**（本期范围 + **本期不做**，本期不做不少于 3 条） | 用户确认或修订 |
| R6 | 历史关联需求（可选；用户跳过则记 N/A 并进入下一步） | 用户确认、跳过或修订 |
| R7 | 行业与竞品（可选；用户跳过则记 N/A） | 用户确认、跳过或修订 |
| R8 | 验收标准与 DoD | 用户确认或修订 |

**引导方式**（每 Step 内仍保持简短）：
- 简短说明 + 与当前 Step 对齐的少量具体问题
- 对业务流程追问主路径与异常分支（主要在 R4）
- 「本期不做」在 R5 强制执行条数要求

**最低可评审标准**（与 R1–R5、R8 对齐）：
- 清晰的用户故事
- 可列出的功能点与优先级
- 明确的本期范围与本期不做
- 可执行的验收标准

### 第三阶段：生成文档

仅在用户选择 **「直接落盘」** 或 **在「预览」后明确确认写入** 后执行：按模板生成 Markdown，保存到 `.ktaicoding/req/`。若用户仅要预览，本轮**只输出**草案内容，不写文件。

### 第四阶段：增量更新

再次执行时：
1. 用户须显式指定待更新的目标文件路径
2. 保留未变更的章节内容
3. 仅更新用户修正的部分
4. 更新文末「变更记录」

## 文档结构

### 标准版章节（/req）

详见 `references/requirement-template-standard.md`。

必须包含：
1. **元信息**：文档编号、创建日期、负责人、状态
2. **一、用户故事**：角色/功能/目的
3. **二、目标用户**：用户类型、描述、核心诉求
4. **三、需求背景**：业务背景、要解决的问题、预期收益
5. **四、需求详细说明**：功能列表（序号、功能点、优先级、说明）、业务流程
6. **五、本期边界**：本期范围、本期不做（重点！）
7. **六、历史相关需求**：关联 REQ 编号、关系、说明
8. **七、行业情况**：业界参考、竞品分析
9. **八、验收要求**：验收标准、完成定义 DoD
10. **九、附录**：变更记录表

### 简化版章节（/req --simple）

详见 `references/requirement-template-simple.md`。

必须包含：
1. **元信息**：文档编号、创建日期、负责人、状态
2. **一、用户故事**：角色/功能/目的
3. **二、功能说明**：功能描述、功能列表（含本期边界）
4. **三、验收标准**：可勾选列表或条目
5. **变更记录**

## 模板覆盖

若项目存在 `.aicoding/templates/overrides/requirement.md`，优先使用该模板。

## 引导话术示例

**用法**：仅在对应 Step（如 R1、R5、R8）的**那一轮**引用与本 Step 匹配的一小段；其余 Step 自行组织短问句。

### 用户故事引导

请描述用户故事，格式为：
"作为一名 [角色]，我希望 [功能]，以便 [目的]"

### 本期边界引导

本期边界是防止范围蔓延的关键。请明确：

**本期范围**：本期要交付的功能/场景

**本期不做**：明确排除的功能/场景

请列出至少 3 条「本期不做」的内容。

### 验收标准引导

验收标准应可量化、可测试。请按以下格式：

- [ ] 功能点 A：预期行为描述
- [ ] 功能点 B：预期行为描述

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| 未提供输入 | 提示：「请提供草稿目录路径或需求文字描述」 |
| 目录不存在 | 提示：「目录 {path} 不存在，请确认路径」 |
| 文件名冲突 | 提示：「文件已存在，请调整序号或名称」 |
| 输出目录不存在 | 创建 `.ktaicoding/req/` 目录 |

## 约束

- 不自动扫描仓库或 `.ktaicoding/req/` 目录
- 不为自动分配序号而扫描目录
- 不实现 `/design`、`/dev` 等其他技能
- 不导出非 Markdown 格式

## 与下游衔接

生成的 REQ 文档路径约定为 `.ktaicoding/req/REQ-{YYYYMMDD}-{序号}-{中文名称}.md`，元信息中的文档编号与文件名一致，便于 `/design` 流程引用。
