# E2E 场景选择规则

## 优先生成 E2E 的场景

- 真实用户可感知的核心页面路径。
- 新增、编辑、删除、搜索、详情、状态流转。
- 权限按钮显隐与只读/可编辑状态。
- Chat、调试、工作台等需要真实交互或流式输出的能力。
- Trace、监控、报告等需要真实数据页面可见性的只读验收。
- 多步骤配置链路：选择应用/版本 → 填表 → 保存 → 列表/详情验证。
- 需求验收标准明确提到页面、按钮、表格、弹窗、输出结果。

## 默认不生成 E2E 的场景

- 纯后端算法、工具函数、ORM migration。
- worker、定时任务、消息消费等无页面入口的内部链路。
- 只能通过 DB/API 判断的逻辑。
- 需要大量稳定测试资产才能可靠验证的复杂组合路径。
- 会产生高风险副作用的操作，例如批量删除、重跑聚合、fork 调试会话，除非 Plan 明确要求且有安全前缀/隔离方案。

## 优先级建议

- P0：新增/编辑/保存/核心 Chat 发送/关键详情打开等阻断主流程的页面路径。
- P1：搜索、筛选、删除清理、Trace 查看、监控图表切换。
- P2：分页、排序、辅助面板、tooltip、非关键只读详情。

## 写操作安全边界

写操作必须满足：

- `fixture_contract` 有唯一测试数据前缀，如 `e2e-<module>-`。
- `cleanup` 明确只通过页面清理该前缀数据。
- 无页面清理入口时标记清理受限或失败，不默认改用接口清理。

## 只读页面验收

Trace、监控、报告、日志类页面通常是只读 E2E：

- 可以使用现有数据做页面可见性断言。
- 不要求创建 fixture。
- 不点击会产生新任务、新会话、新聚合的按钮，除非合同明确授权。
