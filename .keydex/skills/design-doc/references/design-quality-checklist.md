# design-doc 质量校验清单

用于 `/design` 在准备预览或落盘前做内容完整性自检。

目标：避免生成“章节齐全但内容空泛”的 DES。

---

## 一、整体结构检查

- [ ] 文档开头已说明设计目标、范围边界与阅读建议
- [ ] 已识别并标注需求类型：机制型 / 接入型 / 混合型
- [ ] 已提供整体架构图、全链路图、状态图或数据流图中的至少一种
- [ ] 已列出功能点总表，而不是直接进入横向章节平铺
- [ ] 已解释这些功能点如何串联成完整需求效果

---

## 二、功能点详细设计检查

对每个核心功能点，至少确认：

- [ ] 已写清功能目标与边界
- [ ] 已写清触发方式 / 入口
- [ ] 已展开核心逻辑，而不是只有一句话概述
- [ ] 已说明关键流程 / 状态流转 / 生命周期（如适用）
- [ ] 若存在多参与方、异步链路、事件流、锁或状态切换，已补时序图 / 流程图
- [ ] 若存在规则判断、裁剪、聚合、调度、回退、降级逻辑，已补伪代码或结构化流程
- [ ] 若涉及接口，已写清调用方、路径、关键字段与异常场景
- [ ] 若涉及数据设计，已写清表/字段/schema/cache/event 的变更与原因
- [ ] 已写清与现有项目的关联：复用什么、改什么、为什么放在这里
- [ ] 已覆盖关键边界与失败处理

---

## 三、横切设计检查

- [ ] 已有接口设计汇总，且与各功能点详细设计一致
- [ ] 已有数据库 / 模型设计汇总；若涉及数据库表设计，已使用 DDL 写法而不是仅用字段表格
- [ ] 已说明配置变更（如适用）
- [ ] 已说明多服务改动与依赖方向（如适用）
- [ ] 已说明安全 / 权限 / 审计要求（如适用）

---

## 四、项目结合度检查

- [ ] 已明确列出现有可复用能力
- [ ] 已明确设计约束来自哪些现有代码或宪法规则
- [ ] 没有只讲“理想方案”却不讲“当前项目怎么接”的章节
- [ ] 关键模块、接口、数据的放置层级有理由说明

---

## 五、测试与风险检查

- [ ] 测试点按功能点或验收项追溯，不是泛泛列测试类型
- [ ] 至少覆盖正常 / 边界 / 异常三类关键场景
- [ ] 已说明关键技术风险与缓解措施
- [ ] 如有宪法偏离，已说明原因与授权记录

---

## 六、空泛内容检查（强制）

出现以下任一情况，说明文档还不够深，不应直接落盘：

- [ ] 是否还有空表格、占位符、`{说明}`、`TODO` 未替换
- [ ] 是否存在大量“支持 xxx / 用于 xxx / 系统实现 xxx”但没有往下展开逻辑
- [ ] 是否存在应该有图却没有图、应该有伪代码却没有伪代码的章节
- [ ] 是否存在应该落到接口/数据/配置却只停留在概念层的功能点
- [ ] 是否存在与现有项目关联不清、落点不明的设计描述

若上述任一项无法通过，应优先补：
1. 主链路总览
2. 核心功能点详细设计
3. 接口 / 数据 / 项目关联
4. 测试追溯与风险
