# 交互引导与决策话术（增强版）

协作定稿时按需查阅；与 `SKILL.md` 中「协作输出规范」配合使用。

本文件不只用于“问用户选 A/B/C”，还用于：
- 识别需求类型
- 引导用户先建立整体总览
- 按功能点把详细设计写深
- 在落盘前做内容质量校验

---

## 一、关键分歧未收敛时：决策对话模板

```text
【决策点】{一句话：正在决定什么}

| 选项 | 要点 | 与宪法/现有代码 |
|------|------|-----------------|
| A | … | … |
| B | … | … |
| C（可选） | … | … |

【推荐】{默认推荐项}，理由：{1～3 句}
【待你确认】请选择 A/B/C，或补充约束（性能/兼容/上线窗口/ownership 等）。
```

未获用户确认前，不得将某一选项当作已定事实写入拟落盘的 DES 终稿；若必须先写草案，须单列「待确认假设」表。

---

## 二、需求类型识别提示（新增）

用于判断本次 DES 应该偏“机制型”“接入型”还是“混合型”。

```text
我先判断一下这条需求更偏哪一类：

1. 机制型：中间件、调度、压缩、锁、状态机、生命周期、链路编排
2. 接入型：新能力接入、模块扩展、前后端联动、多服务串联
3. 混合型：两者都明显存在

我会重点回答：
- 这条需求的主矛盾是什么？
- 应该先从整体链路讲，还是先从模块接入讲？
- 开篇更适合总览时序图、状态图，还是总体架构/数据流图？
```

### 识别后应调整的文风

- **机制型**：优先画全链路时序 / 状态图，重点讲触发、状态变化、并发和容错。
- **接入型**：优先画总体架构 / 数据流图，重点讲模块职责、接口、前后端协作、存储设计。
- **混合型**：先给整体链路，再在功能点详细设计中分别按机制块与模块块展开。

## 三、整体总览引导（新增）

> 目标：先把“需求整体长什么样”讲清楚，而不是一上来平铺接口和表结构。

```text
在进入详细设计前，我先给你一个总览：

1. 这个需求整体的入口是什么？
2. 从入口到结果，中间会经过哪些关键模块或功能点？
3. 哪些功能点是主链路，哪些是横切支撑？
4. 最终如何形成一个完整的业务/系统效果？

我会先输出：
- 一张整体架构图 / 全链路图
- 一张功能点总表
- 一段核心链路串联说明
```

### 总览写法提醒

总览层至少应回答：
- 用户/系统从哪里进入
- 关键参与方是谁
- 关键数据或状态如何流动
- 哪几个功能点串联成完整需求效果
- 哪些功能点后面需要深入展开

---

## 四、功能点拆分提示（新增）

> 目标：不要只按“接口/数据库/配置”横切，而是先按功能点拆，再把技术细节挂进去。

```text
我先把需求拆成功能点，后续会逐个展开：

| 功能点 | 解决的问题 | 关键参与模块 | 是否需要详细时序 |
|--------|------------|--------------|------------------|
| ... | ... | ... | ... |

拆分原则：
- 先拆主链路功能点
- 再拆关键支撑能力
- 再收口横切设计（接口、库表、配置、测试、风险）
```

### 功能点拆分检查

- 是否覆盖了需求中的主流程
- 是否遗漏了关键状态切换 / 异步链路 / 权限 / 锁 / 回退等重要能力
- 是否把“横切支撑内容”误当成功能点主体
- 是否有必要把一个过大的功能点继续拆小

---

## 五、功能点详细设计提示（核心新增）

> 对已收敛的功能点，应直接展开深入设计，而不是只给一句话概述。

```text
下面进入这个功能点的详细设计，我会按以下顺序展开：

1. 功能目标与边界
2. 触发方式 / 入口
3. 核心逻辑
4. 关键流程 / 状态流转
5. 时序图 / 流程图（如需要）
6. 伪代码 / 关键算法（如需要）
7. 涉及接口设计（如有）
8. 涉及数据结构 / 库表设计（如有）
9. 与现有项目关联
10. 边界与失败处理
11. 测试点
```

### 功能点必须回答的问题

对每个核心功能点，至少回答：
- 它具体解决什么问题？
- 谁来触发？前置条件是什么？
- 主流程分哪几步？
- 涉及哪些现有模块、服务、前端页面、中间件？
- 是否有状态切换、异步行为、锁、并发、幂等等技术点？
- 是否需要新增/修改接口、表、字段、Schema、缓存、事件？
- 如果失败或数据不完整，会怎么处理？

### 详细设计深度提醒

禁止只写这种内容：
- “该功能用于 xxx”
- “系统支持 xxx”
- “后端返回 xxx，前端展示 xxx”

应进一步写到：
- 为什么这样设计
- 关键逻辑怎么串
- 放在当前项目哪个层最合适
- 和现有代码怎么接
- 异常/并发/边界怎么处理

## 六、图与伪代码触发规则（新增）

### 什么情况下应优先给时序图 / 流程图

```text
如果出现以下任一情况，建议直接补图：
1. 参与方 >= 3 且存在明确调用顺序
2. 存在前后端 + 多服务串联
3. 存在异步处理、回调、事件流
4. 存在锁、状态机、生命周期切换
5. 仅靠文字难以快速建立心智模型
```

### 什么情况下应优先给伪代码

```text
如果出现以下任一情况，建议补伪代码：
1. 有规则判断 / 分支裁剪
2. 有聚合 / 调度 / 合并 / 降级逻辑
3. 有状态切换 / 切分 / 回退策略
4. 有“算法味”比较强的机制流程
```

### 若不补图 / 伪代码

应在正文中说明：
- 为什么这里不需要图/伪代码
- 仅用结构化文字已足够表达的原因

## 七、接口与数据库追问提示（增强）

### 接口设计追问

```text
请继续补充这部分接口设计：
- 谁调用谁？
- 是新接口还是复用现有接口？
- 请求体与响应体的关键字段是什么？
- 错误场景怎么表达？
- 是否涉及幂等、权限、分页、筛选、排序、版本解析等约束？
```

### 数据设计追问

```text
请继续补充这部分数据设计：
- 是新增表、改字段，还是只新增 schema / cache / event？
- 若涉及数据库表结构，请直接给出 DDL（CREATE TABLE / ALTER TABLE / CREATE INDEX），不要只列字段表格。
- 关键读写链路是什么？
- 为什么字段/表结构这样设计？
- 是否需要索引、软删除、版本化、回填、迁移？
- 是否存在状态字段、链路字段、指针字段或锁字段？
```

## 八、与现有项目关联提示（新增）

> 目标：避免文档只讲“应该做什么”，却不讲“为什么放在当前项目这里做”。

```text
请说明这部分设计与现有项目的结合方式：
- 复用了哪些现有模块 / 服务 / 页面 / 中间件 / 公共能力？
- 需要修改哪些现有位置？
- 为什么放在这个层，而不是别的层？
- 对现有主链路的侵入面有多大？
- 是否影响其他已有需求或页面？
```

---

## 九、测试点引导（增强）

> 测试点必须追溯到功能点或 REQ 验收项，不能只列“写单测/集成测试”。

```text
请按功能点列测试点：

| 功能点 | 测试场景 | 场景类型 | 预期结果 |
|--------|---------|---------|---------|
| {功能点} | {场景描述} | 正常/边界/异常 | {预期} |

如果需求有明确验收项，请补充：
- 这个测试点追溯的是哪条验收项
- 哪些边界场景必须覆盖
- 哪些失败场景必须验证
```

## 十、宪法一致性自检

```text
请检查设计与项目宪法的一致性：

[ ] 接口路径前缀符合宪法约定
[ ] 请求方式、参数来源与响应格式符合项目规范
[ ] 错误码 / 异常风格与项目约定一致
[ ] 表 / 字段 / schema 命名符合数据库规范
[ ] 多服务依赖方向没有违反架构边界
[ ] 测试策略与项目约定一致

如有偏离，请在“风险与注意事项”中说明原因和授权记录。
```

## 十一、章节质量校验提示（新增强制）

> 在准备落盘前，必须用这一段做内容完整性自检。

```text
请检查当前 DES 是否满足以下质量门槛：

1. 是否已经给出整体总览视图，而不是一上来平铺章节？
2. 是否列出功能点清单，并解释这些功能点如何串联成完整需求效果？
3. 每个核心功能点是否已经详细展开，而不是只有一句概述？
4. 需要时序图 / 流程图 / 伪代码 / 接口 / 数据设计的地方是否缺失？
5. 是否明确写出了与现有项目的关联文件、模块、主链路挂接点？
6. 是否覆盖了关键异常、边界、并发、锁、回退、降级等问题？
7. 测试点是否能追溯到功能点或验收项？
8. 是否仍有空表格、占位句、泛泛描述未被替换？
```

### 质量不达标时的处理方式

- 不要急于落盘
- 回到对应功能点或横切章节继续补深
- 优先补：主链路、接口、数据、异常、测试追溯

## 十二、示例开场话术（可直接复用）

### 示例 1：先总览再细化

```text
我先不急着把章节铺开，先给你一个整体总览：
- 这个需求整体链路怎么走
- 关键参与模块有哪些
- 会拆成哪几个功能点
- 哪几个点需要重点展开时序、接口和数据

等这个总览对齐后，我再逐个功能点往下展开详细设计。
```

### 示例 2：功能点进入详细设计

```text
这个方向已经收敛了，我不再只给决策表，直接进入这个功能点的详细设计：
先讲目标和边界，再讲流程、接口、数据和与现有代码的结合点；如果这里有异步链路或状态切换，我会直接补时序图或伪代码。
```

### 示例 3：准备落盘前的质量检查

```text
在落盘前，我会先做一次内容质量检查，重点看：
- 有没有整体总览
- 功能点有没有逐个展开到技术粒度
- 图、伪代码、接口、数据、测试追溯有没有缺失

如果还停留在概念层，我会先补深，再进入最终成稿。
```
