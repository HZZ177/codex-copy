## 何时使用

编写设计文档、技术/架构/接口/数据库方案、提到 DES、`/design`、协作式设计、把需求落成可开发的设计文档。

若用户只要「评审意见」而不需要 DES 落盘，说明本 skill 主产出仍是 DES 撰写，或请用户改用纯对话完成评审诉求。

# 设计文档撰写（DES）

## 目标与边界

### 主目标

通过 `/design`，将 REQ 与需求要点转化为符合 `references/design-template-standard.md` 的 DES，保存到 `.ktaicoding/des/`，并与项目宪法对齐，供后续 `/dev-plan` 等流程衔接。

DES 的目标不是只把章节填满，而是把需求转成：
- **先看得懂全貌**：需求整体长什么样、功能点如何串起来形成完整效果
- **再看得懂细节**：每个核心功能点如何实现、如何和现有项目结合
- **能直接进入开发**：关键链路、接口、数据、配置、风险、测试点具备足够技术粒度

### 非目标（不做评审动作）

- 不组织、不代替**设计评审会**，不以「评审结论 / 通过与否」为主交付物。
- 不以**批注/打分他人既有稿件**为主任务；若用户仅需评审，说明本 skill 默认仍产出 DES 协作流程，或引导其用普通对话完成评审诉求。
- 不输出「评审会议纪要」作为替代 DES 的正式产物。

### 协作模式（默认）

- **默认**：多轮对话，先收敛关键分歧，再逐步展开深入设计。
- **优先顺序**：先整体总览，再功能点拆解，再横切汇总（接口/数据/配置/测试/风险）。
- **定稿落盘**：仅在满足「可定稿标准」或用户明确要求写文件时，将完整 DES 写入 `.ktaicoding/des/`。
- **例外**：用户**显式**要求「一次性生成完整 DES 草案 / 跳过讨论」时，可一次输出长文，但文首须附 **「已假设的决策」** 表，未决项标为待确认。

## 交互节奏（强制）

- **默认逐步推进，但不要求死板按章节顺序输出**：除用户显式要求一次性出稿外，协作阶段应优先按「整体总览 → 功能点清单 → 功能点详细设计 → 横切收口」推进。
- **关键分歧先决策，已收敛部分可直接展开详细草稿**：
  - 未收敛时：先输出决策点与选项。
  - 已收敛时：允许直接输出当前功能点的详细设计草稿，不必只停留在短摘要。
- **本轮重点只聚焦一个主主题**：可以是一条关键决策，也可以是一个功能点或一组强相关功能点；避免一轮内把整篇 DES 全量定稿。
- **落盘方式（用户选择）**：满足「可定稿标准」或用户主动要求定稿时，先询问：**直接落盘**（写入 `.ktaicoding/des/`）或 **先生成预览**（仅输出 DES 全文或章节摘要草案，用户确认后再写入）。**不得**在未获用户明确选择前写文件。

## 协作输出规范（强制）

### 1. 关键分歧未收敛时

每个决策点（或每一轮）用结构化形式呈现，避免在散文中藏默认选择：

| 元素 | 要求 |
| ---- | ---- |
| 决策点 | 一句话说明在决定什么 |
| 可选方案 | 至少 2 项（可含「维持现状」） |
| 对比 | 各方案优缺点、与宪法及现有代码的符合度 |
| 推荐 | 默认推荐项及理由（可标「待你确认」） |
| 待你确认 | 明确请用户选择或补充约束 |

**禁止**：用户未确认前，用「默认替你选」写入终稿且不标注假设。

### 2. 方向已收敛时

应主动把当前主题写深，不要只给提纲。对核心功能点，至少说明：
- 解决什么问题
- 从哪里触发、涉及哪些参与模块
- 主流程 / 状态流转 / 异步链路
- 是否需要时序图、流程图、伪代码
- 是否涉及接口、数据库、配置或多服务改动
- 若涉及数据库表设计，必须使用 **DDL 写法** 描述表结构与索引（`CREATE TABLE` / `ALTER TABLE` / `CREATE INDEX` 等），不要仅用字段表格替代
- 与现有项目代码如何关联
- 边界、失败处理、测试点

## 输入要求（强制）

需求来源须显式提供，禁止自动扫描推断「要设计什么需求」：

1. **REQ 文件路径**：`.ktaicoding/req/REQ-{YYYYMMDD}-{序号}-{中文名称}.md` 或等价约定路径。
2. **需求要点文字**：无法指向文件时，在对话中给出可与 REQ 对齐的要点。

未提供时：提示补充「REQ 路径」或「可结构化设计的文字要点」，不得猜测或扫仓库推断目标需求。

## 项目代码分析（重要）

需求明确后须扫描分析项目代码，使方案与现有系统融合。必查：项目结构、技术栈（`pom.xml`/`go.mod`/`package.json` 等）、配置、分层、代码风格、数据库与迁移、接口形态、公共组件。按需查：相关模块、测试、中间件。

分析结果不能只停留在“技术栈概览”，还应回答：
- 现有项目中哪些模块可以直接复用
- 哪些链路需要扩展、挂接或替换
- 设计为什么落在这里，而不是别的层
- 关键功能点分别会触达哪些代码边界

复杂项目应在 DES 中加强：
- 「项目现状分析」
- 「现有可复用能力」
- 「与现有项目关联」

## 需求类型识别（新增）

在深入展开详细设计前，应先识别本次需求属于哪一类，并据此调整文风和章节重心：

### 1. 机制型

特征：
- 中间件、调度、压缩、缓存、锁、生命周期、状态机、链路编排

写法要求：
- 开篇优先给**全链路总览时序图 / 状态流转图**
- 详细设计优先写：触发条件、状态变化、并发、容错、生命周期

### 2. 接入型

特征：
- 新能力接入、前后端联动、模块扩展、文件上传、多服务串联

写法要求：
- 开篇优先给**总体架构图 / 数据流总览**
- 详细设计优先写：模块职责、接口、数据流、前后端协作、存储设计

### 3. 混合型

特征：
- 同时有明显的内部机制改造和外部接入改造

写法要求：
- 总览层同时交代架构视图和关键链路
- 详细设计里允许一部分按机制展开，一部分按模块或功能点展开

## 文档命名与输出

- 格式：`DES-{YYYYMMDD}-{序号}-{中文名称}.md`（序号建议与关联 REQ 一致；未指定默认 `001`；名称用安全 slug，限制在10字以内）。
- 路径：`.ktaicoding/des/DES-{YYYYMMDD}-{序号}-{中文名称}.md`。
- 文件名冲突：提示调整序号或名称，禁止静默覆盖。

## 工作流程

### 第一阶段：获取输入

确认 REQ 路径或要点；解析 REQ 元信息；标记空缺项。

### 第二阶段：项目代码分析

扫描结构、技术栈、分层、相关实现、约束与可复用点。空项目可简化，仍须对齐宪法（若存在）。

### 第三阶段：加载宪法与对齐检查

读取 `.ktaicoding/CONSTITUTION.md`；设计不得与生效条文冲突，除非在「风险与注意事项」记录偏离与授权。宪法缺失：建议 `/project-init` 或用户声明临时约定并留痕。REQ 与宪法冲突：列出冲突点与可选方案，待裁决后定稿。

### 第四阶段：协作迭代与设计闭环（常态路径）

对模糊、未闭环或与宪法未对齐的项，分轮引导；每轮遵守「协作输出规范」与「交互节奏」。

默认顺序如下：

| Step | 本轮主题 | 目标 |
| ---- | -------- | ---- |
| G1 | 需求类型识别与整体目标 | 判断机制型 / 接入型 / 混合型，明确文档主结构 |
| G2 | 需求整体总览 | 先建立整体心智模型：整体架构、关键链路、功能点清单 |
| G3 | 功能点拆分 | 把需求拆成若干核心功能点，说明如何串联成完整效果 |
| G4 | 功能点详细设计 | 按功能点逐个展开，写清逻辑、图、伪代码、接口、数据、项目关联 |
| G5 | 横切设计收口 | 汇总接口、数据库、配置、多服务、安全、审计等横向内容 |
| G6 | 测试、风险与宪法一致性 | 做测试追溯、风险说明和最终一致性自检 |

**触发协作**（任一即进入）：
- REQ 关键处模糊
- 功能点边界未闭环
- 接口/数据/配置未收口
- 时序、状态或异常路径还不清楚
- 测试点无法覆盖验收标准

### 第五阶段：章节质量校验（新增强制）

生成文档前，必须进行内容完整性自检，不能只检查“章节是否齐全”。

至少确认：
- 是否已经给出整体总览视图，并解释功能点如何串起来形成完整需求效果
- 是否已经列出功能点清单，且核心功能点都已展开
- 每个核心功能点是否至少写到：逻辑 + 项目关联
- 需要时序图/伪代码/接口/数据库的地方是否已补齐
- 若涉及数据库表设计，是否已经给出 DDL，而不是只停留在字段表格
- 是否存在空表格、纯占位句、只讲概念不落地的章节
- 测试点是否可追溯到功能点或 REQ 验收项

### 第六阶段：生成文档

仅在用户选择 **直接落盘** 或 **预览后确认写入** 后：按模板生成 Markdown，写入 `.ktaicoding/des/`。模板优先级：`.aicoding/templates/overrides/design.md` 若存在则优先于内置 `references/design-template-standard.md`。若用户仅选择预览，本阶段仅输出草案文本，**不**写文件。

### 第七阶段：增量更新

用户显式指定目标 DES 路径；保留未变章节；以最新 REQ 与宪法更新；维护「变更记录」。

## 可定稿标准（升级版）

### 一、可落盘前提

- 关联 REQ 清晰；
- 与宪法无未裁决冲突；
- 核心分歧已收敛；
- 关键功能点边界清晰。

### 二、内容质量门槛

- 已有整体总览视图，且解释了功能点如何串联；
- 已有功能点清单，核心功能点都已详细展开；
- 每个核心功能点至少落到了逻辑 + 项目关联；
- 需要时序/伪代码/接口/数据库的地方没有缺失；
- 若涉及数据库表设计，已使用 DDL 表达而不是只用字段表格；
- 测试点可追溯到功能点或 REQ 验收项；
- 不存在只有空表格/占位句的章节。

未满足：继续协作，或仅产出带「草案」状态与「待确认假设」表的对话内容（仍不写文件，除非用户选了预览且仅预览）。

## 标准章节结构

见 `references/design-template-standard.md`。

标准结构应体现三层：
1. **总览层**：需求整体架构视图、关键链路、功能点清单
2. **详细层**：按功能点展开的详细设计
3. **横切层**：接口汇总、数据库汇总、配置、多服务、测试、风险

不适用章节保留标题并标 N/A 及原因，但不得用 N/A 逃避本应展开的核心设计。

## 错误处理

| 场景 | 处理方式 |
| ---- | -------- |
| 未提供输入 | 提示：请提供 REQ 路径或需求要点描述 |
| REQ 不存在 | 提示路径无效，请确认 |
| 宪法不存在 | 建议 `/project-init` 或声明临时约定 |
| 文件名冲突 | 提示调整序号或名称，禁止覆盖 |
| 输出目录不存在 | 创建 `.ktaicoding/des/` |
| REQ 与宪法冲突 | 列出冲突点，请用户裁决 |

## 约束

- 禁止自动推断需求；禁止为分配序号而扫描 `.ktaicoding/des/`。
- 需求明确后必须分析项目代码。
- 不实现 `/req`、`/dev`、`/cr` 等其他技能；不导出非 Markdown。
- 宪法为硬约束，不得静默忽略。
- 非「评审动作」；主产出为协作迭代后的 DES 落盘。
- 不得只产出概念性提纲；对核心功能点必须给出足够技术粒度。
